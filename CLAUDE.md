# Arduino Card Game — CLAUDE.md

## Обзор проекта

Цифровая карточная игра в сеттинге **Зноя** — постапокалиптического пустынного мира, где одна катастрофа (Зной — день, когда выгорело небо) превратила мир в пустыню и оставила семь несовместимых технологических фракций, идущих к общему призу: уцелевшему в центре пустыни схрону старого мира — **«Колыбели»**. Механика вдохновлена Hearthstone: пошаговый режим, система маны, существа/заклинания/герои (оружие — опциональный модуль). Физическое игровое поле строится на базе RFID-считывателей и RGB-светодиодов.

Этот файл содержит **правила разработки и ядро механики** — то, что нужно при работе над любой картой. Развёрнутый сеттинг, описания фракций и аппаратные конвенции вынесены в `docs/` и читаются по требованию (см. «Справочник»). База знаний открывается как Obsidian-vault.

## Система карточек

Карточки хранятся в `cards/` как отдельные Markdown-файлы.

### Структура папки

```
cards/
  minions/
  spells/
  weapons/
  heroes/
```

### Схема фронтматтера

```yaml
---
id: уникальный-kebab-case-id
name: "Отображаемое имя карты"
type: minion | spell | hero | weapon   # weapon — опциональный модуль (не финализирован)
faction: jackals | ash | chimera | bastion | net | oasis | mirage | neutral
cost: 3
attack: 2        # только для существ (и оружия, если модуль принят)
health: 4        # только для существ и героев
durability: 3    # только для оружия (опциональный модуль)
unit_type: ground | air      # только для существ: наземный | воздушный (по умолчанию ground)
rarity: common | rare | epic | legendary
tags: [тег1, тег2]   # например: [mech, organic, device, drone]
art_prompt: ""   # полный Midjourney-промпт, см. docs/system/art-system.md
art_image: "assets/art/cards/minions/{id}.png"   # путь к PNG-арту (по типу карты); имя файла = id
artist: ""       # необязательно
---
```

После фронтматтера в теле карты — в таком порядке:
1. **Встраивание арта:** `![[assets/art/cards/{тип}/{id}.png]]` — та же строка, что в `art_image`. Когда PNG лежит по этому пути, Obsidian отрисует картинку прямо в заметке; пока файла нет — пустой плейсхолдер.
2. Флейворный текст (1–3 строки).
3. `## Способность`.

**Конвенция арта:** одно изображение на сущность, имя файла = её `id`. Папки: фракции — `assets/art/factions/<id>.png`; карты — `assets/art/cards/<тип>/<id>.png` (`minions` / `spells` / `heroes` / `weapons`). Поле `art_image` и путь во встраивании `![[…]]` должны совпадать. Подробнее — [docs/system/art-system.md](docs/system/art-system.md).

> **Оружие (`type: weapon`) — опциональный модуль, ещё не финализирован.** Ядро игры (существа, заклинания, герои) полно и без него. Поля `durability`/правила экипировки и индикация оружия могут измениться или быть удалены; не опираться на оружие при дизайне фракций (ни одна из 7 фракций на него не завязана). Ячейку героя при принятии модуля можно расширить светодиодами (аппаратного потолка `40` LED там нет).

## Игровая механика

- **Ходы:** игроки чередуются; активный игрок тянет 1 карту, затем разыгрывает действия
- **Право первого хода:** второй игрок начинает с `+1` картой в стартовой руке и получает одноразовый жетон **«Резервная батарея»** — добавляет `+1` ману в любой один свой ход (это не изменение стоимости карты, а разовый ресурс маны)
- **Мана:** начинается с 1, растёт на 1 каждый ход до максимума 10; полностью восстанавливается
- **Поле боя:** до 7 существ на каждой стороне
- **Условие победы:** снизить здоровье вражеского героя до 0
- **Атаки:** существо атакует раз в ход; без **Спешки** — не в ход выставления
- **Оружие героя** *(опциональный модуль, не финализирован — см. примечание в «Системе карточек»)*: если модуль принят — герой может экипировать оружие (`type: weapon`) и атаковать им раз в ход; атака и прочность напечатаны на карте. Атакуя, герой считается **наземным** юнитом (бьёт только наземные цели), если оружие явно не даёт **Досягаемость**. Индикацию остаточной прочности на ячейке героя проектировать при финализации модуля (ячейку можно расширить LED)
- **Усталость (опустошение колоды):** если в начале хода тянуть нечего — герой получает урон усталости, растущий на `+1` за каждый такой добор (`1`, `2`, `3`, …). Гарантирует завершение партии и ограничивает бесконечную ценность контрольных колод

**Универсальные ключевые слова:**

| Слово        | Эффект                                                            |
|--------------|-------------------------------------------------------------------|
| Провокация   | Враги обязаны атаковать это существо (только те, кто способен по типу — см. ниже) |
| Спешка       | Может атаковать в ход выставления                                 |
| Досягаемость | Наземное существо может атаковать **воздушные** цели              |
| Перегрузка X | В следующий ход максимум маны снижается на `X`                    |
| Утилизация   | При гибели: потянуть карту                                        |
| Невидим      | Нельзя атаковать и выбрать целью способностей до первой атаки этого существа; AoE-урон применяется нормально |

**Типы юнитов: наземный / воздушный**

Каждое существо имеет тип `наземный` (по умолчанию) или `воздушный`. Тип напечатан на карте и не меняется, кроме явных эффектов.

Правила атаки:

- **Наземные цели** (наземные существа и **герой**) могут быть атакованы кем угодно.
- **Воздушные цели** могут быть атакованы только **воздушными** существами или существами с ключевым словом **Досягаемость**.
- **Воздушное** существо может атаковать любые цели (и наземные, и воздушные) — оно «достаёт» и землю, и воздух.
- **Наземное** существо без **Досягаемости** бьёт только наземные цели (включая героя). С **Досягаемостью** — также воздушные.
- **Провокация** обязывает атаковать только тех врагов, кто *способен* атаковать данное существо по типу; если враг не может его достать — Провокация его не ограничивает.
- Тип влияет **только на атаку**. Заклинания, способности и AoE-урон выбирают цели по своим правилам, игнорируя тип.
- **Досягаемость** на воздушном существе избыточна и игнорируется.

**Совместимость ключевых слов**

- **Невидим** несовместим с **Провокацией**: невидимого нельзя выбрать целью, поэтому Провокация на него не накладывается (если эффект пытается дать оба — Провокация игнорируется до спадания Невидимости).

> Уникальные механики фракций (Свора, Перегрев, Адаптация, Броня, Взлом, Цветение, Эхо) описаны в файлах `docs/factions/`. Краткая сводка — в [docs/factions/_overview.md](docs/factions/_overview.md).
>
> **Заглушён** — общий статус (наводится Взломом Сети и силами героев): существо теряет текст способностей и все ключевые слова до указанного срока, но в остальном действует нормально (если эффект-источник не запрещает атаку отдельно).

## Правила разработки

### Добавление карточек

1. Выбрать подпапку по типу карты
2. Назвать файл `{id}.md`, совпадает с полем `id`
3. Все обязательные поля заполнены; неприменимые опущены
4. Текст способности — однозначное правило, не лор
5. **Запрещено:** способности, создающие существ на поле вне руки игрока
6. **Запрещено:** любые эффекты изменения стоимости карт — стоимость напечатана на физической карте и является константой

### Принципы баланса

- Существа: `стоимость ≈ (атака + здоровье) / 2`; способности добавляют ~1–2
- **Тип юнита:** `воздушный` ценен как уклонение и гибкость атаки (уязвим только для воздушных и **Досягаемости**, при этом сам бьёт любые цели) — оценивать ~`+1` к стоимости, пока большинство существ не могут бить воздух. `наземный` — нейтральная база. **Досягаемость** на наземном — небольшой плюс ценности (~`+0.5`), растёт с долей воздушных целей в наборе. Оценивать в контексте текущего набора карт
- **Доступность чисток (anti-swarm):** в каждом наборе должны быть надёжные AoE-ответы на go-wide (особенно против **Своры** Шакалов); при дизайне новых широких стратегий проверять, что существует контрмера-чистка. Базовые AoE-носители — заклинания Пепла и Бастиона
- **Свора X** (Шакалы): оценивать при заполненном поле — бонус считать как `+X` к атаке (потолок). Сумма `база_атаки + X` не должна превышать `12` (лимит правой полосы). Тело держать хрупким: ценность Своры в массе, а не в отдельной карте
- **Перегрев** (Пепел): `+1` к атаке в начале каждого хода (видно на правой полосе); **Сброс** наносит цели урон = текущей атаке и сбрасывает атаку к базовой, без ответного урона. Оценивать как растущую угрозу: за `T` ходов Сброс бьёт на `база_атаки + T`. Атака ограничена `12` (предел правой полосы). Закладывать в цену гибкость «бить телом ИЛИ сбросить точечно по любой цели, включая героя» (~+1.5–2 к стоимости при низкой базе)
- **Адаптация** (Химеры): пакет «выбор из 3 усилений» считать как +1–1.5 к стоимости; усиление всегда **выбирает игрок** (управляемость заложена в цену). Условие «пережить урон» — встроенный сдерживающий фактор: не давать Адаптацию телам с `1` здоровьем без компенсации
- **Броня X** (Бастион): `1` Броня ≈ `0.7` здоровья (блокирует урон, но не восстанавливается и не лечится обычным лечением). Броня показана янтарными делениями на левой полосе поверх здоровья, поэтому сумма `здоровье + Броня` ≤ `12` (предел левой полосы) — это и есть потолок выживаемости одного тела
- **Взлом** (Сеть): отключение существа на `1` круг оценивать как темповое удаление силой ниже полного уничтожения (~`-1` к стоимости относительно «уничтожить»); цель возвращается живой. Следить, чтобы у фракции не было дешёвого многократного Взлома (лок-аут)
- **Цветение X** (Оазис): оценивать при заполненном поле — эффективное лечение `≈ X × число дружественных существ` за ход. Несколько Цветений складываются: проверять, что связка не делает фланг непробиваемым против неагрессивных колод
- **Эхо** (Мираж): считать способность существа как ×2 от исходного эффекта. Применять только к одиночным/случайным эффектам и активным/триггерным способностям, не к постоянным пассивкам. Эхо не запускает само себя (без цепочек)
- Легендарные: одна в колоде, уникальная именная способность

### LED-проектирование способностей

При создании новой способности указывать:
- Какая полоса отображает состояние
- Цвет и паттерн анимации
- Как изменяется при изменении стека

Не проектировать механики, чьё состояние невозможно выразить доступными светодиодами. **Принцип: все живые числовые значения эффектов должны читаться с поля, а не храниться скрыто в контроллере.** Для существа доступны **три полосы**: левая `12` — здоровье (и **Броня** — янтарными делениями поверх здоровья; `здоровье + Броня ≤ 12`), правая `12` — атака (включая бонус **Своры** и накал **Перегрева** — всегда показывает текущее значение), верхняя `8` — ключевые слова/статусы (по LED-флагу на позицию). **Нижняя полоса (`8` LED) зарезервирована** и в текущей версии не несёт функции (назначение будет определено позже) — новые механики проектировать так, чтобы их значение умещалось на левой/правой/верхней полосах. Для героя — отдельная раскладка: полоса здоровья (`30` LED, `1` LED = `1` HP) и полоса маны (`10` LED, `1` LED = `1` мана). Полная карта полос, цветов и анимаций — [docs/hardware/led-conventions.md](docs/hardware/led-conventions.md).

### Соглашения по коду

- Парсинг карточек: YAML-фронтматтер, валидация по схеме, отклонять неизвестные поля
- ID в формате kebab-case, глобально уникальны, не переиспользуются
- Никаких захардкоженных данных в движке — всегда загружать из `cards/`
- Тесты: отсутствие обязательных полей, неизвестные поля, некорректные значения

### Контекст для Claude

- Тон карточек: постапокалиптический, технически точный, атмосферный
- Флейворный текст — от первого или третьего лица, 1–3 строки
- Ссылки на электронику точные (реальные компоненты: ATmega328, ESP32, NRF24L01, HC-SR04)
- При балансировке — явно считать соотношение маны к характеристикам
- При проектировании способности — описывать LED-отображение
- Игра пошаговая, однопоточная; нет асинхронных эффектов
- **Жирный** для ключевых слов, `код` для чисел там, где возможна неоднозначность
